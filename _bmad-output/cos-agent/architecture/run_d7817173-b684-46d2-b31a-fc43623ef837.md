## 1. Architecture Overview

Talkin AI should be implemented as a single-user, desktop-first, local-first application with one clear rule: every Korean request becomes durable local state before any local optimization or paid cloud inference begins. That rule is what makes the PRD promises around original-text preservation, error recovery, history detail, and screen-to-screen sync enforceable instead of aspirational.

### 1.1 Chosen baseline stack

| Layer | Choice | Why this baseline fits the PRD |
| --- | --- | --- |
| Desktop shell | Electron + TypeScript | Fastest path for a desktop app that needs secure secrets, child-process control, local file access, and multiple AI provider integrations from one runtime. |
| Renderer UI | React + Vite + TypeScript | Supports dense desktop workspace layouts, strong component reuse across Chat, Workbench, Projects, History, and Settings. |
| Routing | React Router | Keeps screen flows explicit, lightweight, and easy to align with the required desktop navigation model. |
| UI state | TanStack Query for server-like local reads, Zustand for ephemeral UI state | Separates durable domain state from view-only concerns like panel focus, filters, and expanded history rows. |
| Desktop data store | SQLite | Best fit for single-user persistence, restart recovery, local analytics, and projection queries across multiple screens. |
| Secrets | OS keychain via Electron-safe secret API | API keys must not live in renderer state or plain SQLite rows. |
| Local optimization boundary | Translation MCP sidecar managed by the main process | Preserves the product promise that Korean-English optimization and English-Korean restoration stay local and provider-independent. |
| Cloud model integration | Provider adapter layer for OpenAI, Anthropic, and Gemini in the main process | Keeps model calls behind a single contract and avoids renderer-side credential leakage. |

### 1.2 Why Electron is the preferred shell

Electron is the pragmatic first choice because this product needs all of the following on day one: secure API key storage, background workflow execution, child-process lifecycle management for the local Translation MCP runtime, SQLite access, and provider SDK or HTTPS integrations. Tauri is viable later, but starting with Tauri would push complexity into Rust-side service orchestration before the core product loop is proven.

### 1.3 High-level runtime shape

```text
Renderer UI
  Chat / Workbench / Projects / History / Settings
        |
        | typed IPC
        v
Electron Main Process
  Workflow Orchestrator
  Query/Command Handlers
  Event Broadcast
        |                |                    |
        |                |                    |
        v                v                    v
   SQLite Store   Translation MCP Sidecar   Cloud Provider Adapters
        |
        v
  Usage Ledger / Projections / Resume Queue
```

### 1.4 Canonical request lifecycle

1. The user submits Korean text from Chat or Workbench.
2. The main process writes the task, message, and a new run record to SQLite before any model work starts.
3. The orchestrator estimates baseline Korean token cost for the selected cloud model.
4. The orchestrator calls the local Translation MCP sidecar to generate an optimized English prompt for the selected optimization mode.
5. The optimized prompt is validated for required preservation fields such as named entities, constraints, and requested output format.
6. The orchestrator estimates optimized prompt token cost.
7. The selected cloud adapter sends only the optimized English prompt to the cloud model.
8. The English model response is stored as an internal artifact and sent to the local Translation MCP sidecar for Korean restoration.
9. The restored Korean response, usage metadata, latency, savings metrics, and run outcome are committed transactionally.
10. The main process emits change events so Chat, Workbench, Kanban, Project detail, and History projections refresh from the same source of truth.

### 1.5 Architectural decisions fixed in this phase

1. The desktop app is greenfield and should start with Electron, React, TypeScript, and SQLite.
2. The main process is the only runtime allowed to call outbound cloud APIs or touch secrets.
3. Translation MCP is treated as a local sidecar contract, not a UI concern.
4. SQLite is the only durable source of truth for tasks, conversations, history artifacts, settings, and cost records.
5. A task is the canonical work item across Chat, Workbench, Kanban, Projects, and History.
6. Full Korean token-by-token streaming is deferred for v1 because reliable restoration and structure preservation matter more than optimistic partial output.

### 1.6 Recommended codebase shape

```text
apps/
  desktop/
    src/main/
    src/preload/
    src/renderer/
packages/
  domain/
  persistence/
  workflows/
  translation-mcp-client/
  provider-gateways/
  ui-shared/
```

This keeps the desktop shell, domain contracts, orchestration logic, and provider integrations separable without forcing a large monorepo before the first slice exists.

## 2. Major Components

### 2.1 Desktop shell and security boundary

**Responsibilities**

- Boot the window, preload, and typed IPC surface.
- Keep `contextIsolation` enabled and `nodeIntegration` disabled.
- Own all secrets, database access, filesystem access, and provider credentials.
- Launch and monitor the Translation MCP sidecar process.

**Constraints**

- The renderer cannot call providers directly.
- API keys never pass into React props, local storage, or SQLite plaintext tables.
- All filesystem writes go under app-managed local directories.

### 2.2 Renderer workspace application

**Responsibilities**

- Render the chat-first inbox experience as the default first screen.
- Provide the five primary navigation destinations required by the experience brief: Chat, Workbench, Projects, Usage/History, Settings.
- Surface task state, run state, errors, savings, and history artifacts without becoming a developer console.
- Support a dense desktop layout with a left navigation rail and a workspace body that can switch between focused and multi-panel views.

**Constraints**

- The first screen must be a working inbox/composer layout, not a hero landing page.
- Workbench must render at least two live task panels simultaneously on supported desktop widths, with a 2x2 target layout at wider sizes.
- The advanced optimized-English preview is hidden by default and revealed only by explicit user action.

### 2.3 Workflow orchestrator

**Responsibilities**

- Accept commands such as `submitPrompt`, `retryRun`, `moveTaskStatus`, `openInWorkbench`, and `updateSettings`.
- Drive the lifecycle `queued -> optimizing -> optimized -> cloud_pending -> restoring -> completed | failed`.
- Persist stage transitions so the app can recover after restart or crash.
- Prevent duplicate paid retries after ambiguous cloud failures.

**Constraints**

- The Korean source message must be stored before optimization starts.
- A run cannot skip the optimization stage and go directly to the cloud path in normal mode.
- If optimization fails, the app should surface a recoverable error instead of silently sending the raw Korean prompt to a paid cloud model.

### 2.4 Translation MCP client and local optimization engine

**Responsibilities**

- Provide a single main-process client for the local Translation MCP sidecar.
- Support at least these app-facing operations:
  - `optimizePrompt`
  - `restoreResponse`
  - `summarizeConversationContext`
  - `healthCheck`
- Inject optimization-mode strategy instructions for `기본`, `절감 우선`, `품질 우선`, and `긴 컨텍스트`.
- Preserve structural output requirements such as tables, lists, headings, and numeric values.

**Constraints**

- The sidecar must run locally and must not proxy through paid cloud translation APIs.
- The sidecar contract must be provider-agnostic so the cloud gateway layer can evolve independently.
- The app must store both the source Korean prompt and the optimized English prompt for history inspection, while keeping the optimized prompt out of the default chat view.

### 2.5 Cloud model gateway

**Responsibilities**

- Expose a unified `infer()` contract for OpenAI, Anthropic, and Gemini adapters.
- Translate app-level settings into provider-specific request payloads.
- Record provider-reported token counts when available.
- Normalize failures into user-facing error categories such as `auth`, `rate_limit`, `network`, `provider_unavailable`, and `invalid_request`.

**Constraints**

- The cloud gateway only receives the optimized English prompt plus required system/context packaging.
- Provider adapters must not write directly to UI state; they only return structured results to the orchestrator.
- The initial release should support one configured default model at a time per user settings, even if multiple providers are connected.

### 2.6 Usage and cost ledger

**Responsibilities**

- Estimate the token count for the original Korean prompt under the selected target model.
- Estimate the token count for the optimized English prompt under the same target model.
- Store actual provider token usage if the provider returns it.
- Compute per-run and aggregate savings for History and Dashboard views.

**Constraints**

- Savings must compare like-for-like against the same provider/model pricing basis.
- Every usage record must store the pricing version used for the calculation.
- If a provider tokenizer is unavailable, the UI must label the result as an estimate instead of implying billing-grade precision.

### 2.7 Persistence, projections, and sync

**Responsibilities**

- Keep normalized domain data in SQLite.
- Expose read models for Chat, Workbench, Kanban, Projects, and History.
- Broadcast entity-level invalidation events after each committed transaction.
- Resume or safely mark interrupted runs after app restart.

**Constraints**

- SQLite is the system of record. Renderer caches are disposable.
- Task status, project linkage, and conversation history must never be duplicated in screen-specific tables.
- Workbench panel placement is layout state, not domain truth.

### 2.8 Settings and secret management

**Responsibilities**

- Persist non-secret preferences such as theme, response language, default cloud model, optimization mode, and advanced-preview visibility.
- Store provider API keys and local engine credentials only through OS-backed secure storage.
- Expose connection health to the Settings screen.

**Constraints**

- A disconnected provider or local engine must produce a human-readable setup or recovery state.
- The app must not block the whole UI because one provider is misconfigured.

### 2.9 Optional document/file subsystem

**Responsibilities**

- Reserve a place for project-linked files and later local document ingestion.
- Allow project detail to list related local files even before full document parsing is implemented.

**Constraints**

- Imported files should be copied or referenced under app-managed storage with metadata rows in SQLite.
- Deep file parsing, OCR, and semantic indexing are out of the critical path for the first working architecture slice.

## 3. Data, State, or Persistence Concerns

### 3.1 Canonical domain model

| Entity | Purpose | Key fields |
| --- | --- | --- |
| `tasks` | Canonical work item across all screens | `id`, `title`, `status`, `project_id`, `source_screen`, `created_at`, `updated_at`, `last_activity_at` |
| `conversations` | One task’s chat thread | `id`, `task_id`, `summary`, `mode`, `selected_model` |
| `messages` | User-visible Korean turns | `id`, `conversation_id`, `role`, `content_ko`, `run_id`, `created_at` |
| `run_records` | One execution attempt for a user turn | `id`, `task_id`, `conversation_id`, `message_id`, `status`, `provider`, `model`, `mode`, `started_at`, `ended_at`, `error_code` |
| `run_stages` | Fine-grained progress and recovery | `id`, `run_id`, `stage`, `status`, `started_at`, `ended_at`, `details_json` |
| `prompt_artifacts` | Hidden-but-auditable intermediate artifacts | `id`, `run_id`, `artifact_type`, `content`, `token_estimate`, `visibility` |
| `usage_records` | Token, latency, and savings ledger | `id`, `run_id`, `baseline_input_tokens`, `optimized_input_tokens`, `output_tokens`, `estimated_cost_without_optimization`, `estimated_cost_with_optimization`, `pricing_version`, `latency_ms` |
| `projects` | Long-lived work containers | `id`, `name`, `description`, `goal`, `created_at`, `updated_at` |
| `workbench_layouts` | Saved multi-panel view state | `id`, `name`, `created_at`, `updated_at` |
| `workbench_panels` | Which task is open in which panel | `id`, `layout_id`, `panel_slot`, `task_id`, `pinned`, `updated_at` |
| `file_assets` | Project-linked local files | `id`, `project_id`, `display_name`, `storage_path`, `mime_type`, `size_bytes` |
| `settings` | Non-secret user preferences | `key`, `value_json`, `updated_at` |

### 3.2 Persistence rules that stories must preserve

1. Save the Korean source prompt before any optimization or provider work.
2. Store the optimized English prompt and raw English model response as first-class artifacts tied to the same run.
3. Write run-stage updates as separate rows so loading and error states survive restart.
4. Commit restored Korean output and usage data in the same transaction that marks the run complete.
5. Keep task status and project linkage in canonical tables so Chat, Workbench, Kanban, and Projects cannot drift.

### 3.3 Restart and failure recovery

- If the app closes before any cloud request is dispatched, the orchestrator may auto-resume the queued or optimizing run on restart.
- If the app closes after a cloud request may already have been billed, mark the run `interrupted_after_dispatch` and require explicit user retry to avoid accidental duplicate charges.
- Failed runs keep the original Korean message, current stage, error code, and recovery hints so the user can retry without retyping.

### 3.4 Query and sync model

- The main process exposes explicit read queries such as `getChatFeed`, `getWorkbenchLayout`, `getBoardColumns`, `getProjectDetail`, `getUsageDashboard`, and `getHistoryEntry`.
- The renderer uses query invalidation rather than screen-local mutation as the normal refresh path.
- UI-only state such as panel focus, expanded accordions, draft composer text, and view filters stays in renderer stores and is not treated as canonical domain state.

### 3.5 File and disk layout

- SQLite database: app data directory, for example `.../TalkinAI/app.db`
- Files: `.../TalkinAI/files/`
- Logs and diagnostics: `.../TalkinAI/logs/`
- Cached temporary artifacts: `.../TalkinAI/tmp/`

## 4. Integration and Runtime Constraints

### 4.1 Translation MCP runtime contract

The app should speak to a local Translation MCP sidecar through a thin adapter owned by the main process. The app-facing contract should remain stable even if the underlying local runtime changes from user-installed Ollama to an embedded `llama.cpp` bundle later.

Minimum app-facing contract:

```ts
type OptimizePromptInput = {
  sourceKorean: string
  mode: "default" | "cost_saver" | "quality" | "long_context"
  conversationSummary?: string
  outputHints?: string[]
  namedEntities?: string[]
}

type OptimizePromptResult = {
  optimizedEnglish: string
  preservationChecks: {
    entitiesPreserved: boolean
    constraintsPreserved: boolean
    outputFormatPreserved: boolean
  }
}

type RestoreResponseInput = {
  sourceKorean: string
  optimizedEnglish: string
  cloudEnglishResponse: string
  outputHints?: string[]
}

type RestoreResponseResult = {
  restoredKorean: string
}
```

### 4.2 Local engine packaging tradeoff

The first implementation should target a pluggable local runtime adapter instead of hard-wiring one bundled model. That gives the team a fast path with a user-installed local runtime while preserving a future move to a bundled model if onboarding friction becomes unacceptable.

### 4.3 Cloud provider adapter contract

Each provider adapter should implement the same app-level response shape:

```ts
type CloudInferenceResult = {
  provider: "openai" | "anthropic" | "gemini"
  model: string
  englishResponse: string
  usage?: {
    inputTokens?: number
    outputTokens?: number
  }
  requestId?: string
  latencyMs: number
}
```

Provider-specific details such as headers, SDKs, retry semantics, and usage fields stay inside the adapter.

### 4.4 Token and pricing constraints

1. Baseline and optimized token comparisons must use the same target model basis.
2. Savings must be reported as estimated unless provider-billing-grade token usage is actually available.
3. Pricing tables must be versioned so historical cost comparisons remain reproducible when provider prices change later.
4. Dashboard monthly totals should be derived from persisted per-run records, not recomputed from ephemeral UI state.

### 4.5 UI and layout constraints from the experience brief

1. Default route is Chat, not Dashboard.
2. Global navigation must include Chat, Workbench, Projects, Usage/History, and Settings.
3. Kanban should exist as a secondary work-management route or tab, not as the only task view.
4. The first screen silhouette must read like a chat-first inbox with a live composer, suggested work starters, and immediate action context.
5. Workbench should target a 2x2 panel grid at wide desktop widths and must show at least two active panels on supported widths.
6. History detail must support collapsed advanced artifacts so the default experience remains simple while power users can inspect the optimization chain.

### 4.6 Security and privacy constraints

1. Secrets belong in OS keychain storage only.
2. Renderer code never receives raw provider secrets.
3. Outbound network calls are limited to explicitly configured cloud model providers.
4. The Translation MCP sidecar should communicate over stdio or another local-only channel, not a public network interface by default.
5. Logs must redact API keys and should avoid dumping full prompt contents unless a user explicitly enables diagnostic mode later.

### 4.7 Observability and supportability

- Record structured local logs for each run stage and external dependency check.
- Keep a human-readable error mapping layer so Settings and task failures can suggest the next safe action.
- Expose simple health states for local engine, provider connections, and database availability.

## 5. Architecture Constraints for Story Implementation

These are architecture-level constraints that downstream stories should inherit without reopening the core design.

### 5.1 Chat and first-run experience

- Chat is the default entry route and must work before Workbench, Board, or Project features exist.
- The first screen must combine welcome copy, suggested tasks, composer, model selector, and optimization mode selector in one working surface.
- Story acceptance should explicitly reject centered marketing-hero layouts or generic card dashboards as “done.”

### 5.2 Conversation pipeline stories

- Any prompt submission story must prove that the Korean source message is written durably before optimization starts.
- Any optimization story must prove that a failed local optimization does not silently downgrade into a paid raw-Korean cloud request.
- Any response-restoration story must preserve structural content such as lists, tables, numbers, and checklists.

### 5.3 Workbench and Kanban stories

- Workbench stories must reuse the same `task` and `conversation` entities as Chat.
- Kanban stories may only mutate `task.status`; they must not invent a second workflow-state store.
- `작업대에서 계속하기` must open the existing task in a panel instead of cloning a new conversation by default.

### 5.4 History and dashboard stories

- History stories must read from stored run artifacts, not reconstructed text.
- Dashboard stories must show both “without local optimization” and “with local optimization” views from the same persisted pricing basis.
- Advanced English prompt preview must be gated behind explicit user action and setting state.

### 5.5 Settings and recovery stories

- Settings stories must separate secret writes from normal preference writes.
- Error-handling stories must show the failed stage and the next safe action.
- Restart-recovery stories must prove that incomplete runs remain visible and recoverable.

### 5.6 Validation hooks

1. A first-time user should be able to explain the product purpose and first action by looking only at the Chat route.
2. A single task should remain identifiable by the same title, status, and history across Chat, Workbench, Kanban, and Projects.
3. A failed run should be retryable without re-entering the Korean prompt.
4. A power user should be able to inspect source Korean, optimized English, final Korean, tokens, and savings for one run without exposing that detail in the default view.

### 5.7 Must-avoid notes

1. Do not put provider SDK calls in renderer components.
2. Do not store task truth separately per screen.
3. Do not send raw Korean to the cloud as a hidden fallback when local optimization fails.
4. Do not compute savings from mixed providers, mixed models, or unversioned price tables.
5. Do not design Workbench as a tabbed single-chat view and call it multi-chat.
6. Do not make the first screen a marketing page and defer the real composer to a later click.

## 6. Implementation Risks and Tradeoffs

### 6.1 Local runtime onboarding vs product simplicity

Using a pluggable local runtime adapter is the fastest implementation path, but it may add setup friction if the user must install or configure a local model runtime. Bundling a model later improves onboarding but increases app size, update complexity, and hardware variability risk.

### 6.2 No full streaming in v1 vs perceived responsiveness

Deferring live Korean streaming simplifies structural restoration and reduces partial-translation quality problems. The tradeoff is that users may perceive responses as slower than ChatGPT-style streaming. The loading state must therefore clearly show `로컬 최적화`, `모델 응답 대기`, and `한국어 복원` progress so the app never looks frozen.

### 6.3 Multi-provider flexibility vs consistent behavior

Supporting OpenAI, Anthropic, and Gemini behind one gateway improves reach, but usage accounting, response structure, and failure semantics will differ. The adapter layer and price-version ledger are required to keep the UX consistent enough for the Dashboard and History claims to remain trustworthy.

### 6.4 Auditability vs sensitive artifact storage

Persisting optimized English prompts and raw English responses is necessary for History detail, savings evidence, and recovery. The tradeoff is a larger sensitive-data footprint on disk. This makes local filesystem discipline and future encrypted-at-rest options relevant even if they are not mandatory in the first slice.

### 6.5 Scope breadth vs delivery sequence

Chat, Workbench, Kanban, Projects, History, and Settings all rely on the same core task/run model. If the team starts with screen-first implementations before the task and run schema are stable, state drift and rework will follow. The correct sequence is domain model and orchestration first, then screen projections on top.

### 6.6 File management depth vs initial velocity

Project-linked files are part of the intended workspace story, but deep parsing and ingestion will slow the first working release. The architecture therefore reserves a file subsystem without making rich document understanding a day-one dependency for the Korean optimization loop.

## 8. Deferred Decisions to Carry Forward

1. Which local runtime ships first behind the Translation MCP adapter: user-installed runtime, bundled runtime, or both.
2. Which cloud provider and model ship as the initial default preset.
3. Whether project file import stays metadata-only in v1 or includes local text extraction for PDF workflows.
4. Whether Kanban is exposed as a dedicated route or a secondary Workbench/Projects view, while still satisfying the PRD workflow needs.

## 9. Architecture Acceptance for This Phase

This architecture is complete for the phase when all of the following remain true:

1. The document gives implementation stories fixed runtime boundaries: renderer, main process, Translation MCP sidecar, provider adapters, SQLite, and keychain secrets.
2. The document preserves the PRD promises around Korean-first UX, local optimization, savings visibility, desktop workspace expansion, and restart-safe persistence.
3. The document encodes explicit tradeoffs instead of hiding them behind future assumptions.
4. The document is detailed enough for downstream stories to build slices without re-deciding the system of record, security boundary, or request lifecycle.
