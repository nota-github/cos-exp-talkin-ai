## 1. Product Summary

Talkin AI는 한국어 사용자가 영어를 직접 쓰지 않고도 더 낮은 토큰 비용으로 클라우드 AI를 오래, 깊게 사용할 수 있게 만드는 데스크탑 AI 작업 공간이다. 사용자는 채팅처럼 한국어로 바로 시작하고, 앱 내부의 로컬 LLM이 한국어 입력을 영어 기반 토큰 흐름으로 최적화해 클라우드 모델에 전달한다. 클라우드 모델의 영어 응답은 다시 로컬에서 자연스러운 한국어로 복원된다.

초기 제품의 핵심은 번역 기능 자체가 아니라, 한국어 사용 경험을 유지한 채 비용 절감과 장기 작업 관리까지 한 제품 안에서 연결하는 것이다. 따라서 첫 화면은 마케팅 페이지가 아니라 실제 업무를 시작하는 채팅 중심 인박스여야 하며, 긴 작업은 작업대, 칸반, 프로젝트, 히스토리로 자연스럽게 확장되어야 한다.

## 2. Goals

1. 사용자가 처음부터 끝까지 한국어만으로 AI 작업을 시작, 진행, 확장할 수 있게 한다.
2. 로컬 LLM 기반 비용 최적화 엔진이 유료 번역 API 없이 한국어 입력을 영어 기반 토큰 흐름으로 압축하고, 영어 응답을 자연스러운 한국어로 복원하게 한다.
3. 요청별 절감률과 누적 절감 토큰 및 비용을 사용자에게 직관적으로 보여줘 제품의 핵심 가치를 눈에 보이게 만든다.
4. 단일 채팅을 넘어서 여러 장기 작업을 동시에 관리할 수 있는 데스크탑 워크스페이스를 제공한다.
5. 채팅, 작업대, 칸반, 프로젝트, 설정, 히스토리 간 상태가 끊기지 않고 유지되는 신뢰 가능한 작업 환경을 제공한다.

## 3. Non-Goals

1. 이 단계에서 특정 로컬 LLM, Translation MCP 세부 계약, 저장소 구조, 데이터 모델, 런타임 아키텍처를 확정하지 않는다.
2. 모바일 우선 UX나 작은 화면 중심 인터랙션을 설계하지 않는다.
3. 범용 에이전트 마켓플레이스, 외부 툴 생태계 확장, 자동화 플랫폼 역할까지 포함하지 않는다.
4. 협업 권한 체계, 조직 관리, 결제/구독, 운영 콘솔을 이번 PRD의 필수 범위로 넣지 않는다.
5. 마케팅 랜딩 페이지를 핵심 결과물로 취급하지 않는다.

## 4. Target Users and Jobs

### Primary Users

- 한국어로 장문 지식 작업을 반복하는 운영 담당자
- 문서 정리, 요약, 초안 작성, 카피 개선 등 반복형 AI 작업을 수행하는 일반 지식 작업자

### Primary Jobs To Be Done

1. 한국어로 바로 요청을 입력하고 빠르게 결과를 받는다.
2. 한 번의 응답으로 끝나지 않는 작업을 여러 패널과 상태 보드에서 이어간다.
3. 같은 작업의 원문, 최적화 프롬프트, 최종 응답, 절감 효과를 필요할 때 확인한다.
4. 작업이 길어져도 대화와 상태를 잃지 않고 프로젝트 단위로 관리한다.

## 5. Product Principles

1. 기본 언어는 한국어다. 사용자는 영어 프롬프트 작성 의무를 느끼면 안 된다.
2. 비용 최적화는 자동이어야 한다. 일반 모드에서는 내부 번역/압축 과정을 의식하지 않아도 된다.
3. 절감 효과는 이해하기 쉬운 사용자 언어로 보여야 한다. 예: `이번 요청에서 39% 절감`.
4. 첫 화면은 채팅 중심 인박스여야 한다. 차가운 SaaS 대시보드나 랜딩 페이지처럼 보이면 안 된다.
5. 긴 작업은 단일 스레드에 묻히지 않고 작업대, 칸반, 프로젝트로 확장되어야 한다.
6. 데스크탑 앱답게 정보 밀도와 병렬 작업성이 높아야 한다.
7. 오류가 나도 사용자의 입력과 작업 상태는 사라지지 않아야 한다.

## 6. Core User Flows

### Flow A. 첫 진입과 첫 한국어 요청

1. 사용자가 앱을 열면 글로벌 내비게이션과 채팅 중심 인박스가 보인다.
2. 첫 화면은 제품 목적과 첫 행동을 함께 설명한다.
3. 사용자는 추천 작업 카드를 클릭하거나 직접 한국어 프롬프트를 입력한다.
4. 사용자는 기본 모델과 최적화 모드를 확인하고 요청을 전송한다.
5. 앱은 전송 상태를 명확히 보여주고, 요청이 작업 히스토리에 남는다.

### Flow B. 로컬 최적화와 응답 루프

1. 한국어 원문 입력은 대화 히스토리에 그대로 저장된다.
2. 로컬 LLM이 원문을 영어 기반 최적화 프롬프트로 변환한다.
3. 선택된 클라우드 모델이 최적화된 영어 프롬프트로 추론한다.
4. 영어 응답은 로컬 LLM을 통해 자연스러운 한국어로 복원된다.
5. 최종 응답에는 사용 모델, 지연 시간, 절감률 같은 메타 정보가 함께 보인다.

### Flow C. 긴 작업을 작업대로 확장

1. 사용자는 채팅 중인 작업을 `작업대에서 계속하기`로 전환한다.
2. 작업대에서 같은 작업이 특정 패널에 열리고 기존 대화와 상태가 이어진다.
3. 사용자는 다른 작업도 추가해 여러 패널에서 병렬로 진행한다.
4. 각 패널은 독립적인 채팅, 상태, 활동 로그, 추가 지시 입력을 가진다.

### Flow D. 작업 흐름과 프로젝트 관리

1. 사용자는 작업을 칸반 카드로 보고 상태를 바꾼다.
2. 사용자는 작업을 특정 프로젝트에 연결한다.
3. 프로젝트 화면에서 관련 작업, 파일, 문서, 대화 컨텍스트를 함께 본다.
4. 작업 상태 변경은 채팅, 작업대, 칸반, 프로젝트에 동기화된다.

### Flow E. 절감 효과 확인과 상세 히스토리 검토

1. 사용자는 사용량 대시보드에서 이번 달 사용량, 절감률, 절감된 금액, 토큰 감소량을 확인한다.
2. 사용자는 `로컬 최적화 없이 사용했다면` 대비 `로컬 최적화 적용 후`를 비교한다.
3. 사용자는 히스토리 상세에서 원문 한국어 프롬프트, 최적화 영어 프롬프트, 최종 한국어 응답을 단계별 토큰 수와 함께 펼쳐본다.
4. 일반 사용자는 고급 미리보기를 끄고 최종 응답만 볼 수 있다.

### Flow F. 설정과 안정성 회복

1. 사용자는 기본 클라우드 모델, API 키 연결 상태, 로컬 최적화 모드, 응답 언어, 테마를 설정한다.
2. 모델 호출 실패나 로컬 최적화 실패가 발생하면 사용자에게 원인과 다음 안전한 행동이 보인다.
3. 앱 재실행 후에도 대화, 작업, 프로젝트, 설정이 유지된다.

## 7. Functional Requirements

### FR-01. 채팅 중심 인박스 시작 경험

- 첫 화면은 마케팅 랜딩이 아니라 실제 업무 시작 화면이어야 한다.
- 글로벌 내비게이션에 채팅, 작업대, 프로젝트, 사용량 대시보드/히스토리, 설정이 보여야 한다.
- 첫 진입 화면에는 `무엇이든 한국어로 편하게 물어보세요`에 해당하는 환영 메시지와 추천 작업 카드가 있어야 한다.
- 화면 실루엣은 단순 중앙 입력창이나 기본 카드 대시보드와 구별되는 채팅 중심 인박스 방향을 보여야 한다.
- 기본 CTA는 제품 맥락에 맞는 한국어 요청 시작 행동이어야 한다.

### FR-02. 한국어 채팅 입력과 히스토리 보존

- 사용자는 한국어 프롬프트를 입력하고 전송할 수 있어야 한다.
- 입력한 한국어 원문은 수정 없이 대화 히스토리에 보존되어야 한다.
- 긴 한국어 문장, PDF 요약 요청, 보고서 작성 요청도 깨지지 않고 처리되어야 한다.
- 빈 상태는 사용자가 무엇을 먼저 해야 하는지 알려야 한다.
- 로딩 상태는 사용자가 지금 어떤 단계에 있는지 이해할 수 있게 해야 한다.

### FR-03. 로컬 LLM 비용 최적화 엔진

- 클라우드 모델 호출 전 로컬 LLM이 한국어 입력을 영어 기반 최적화 프롬프트로 변환해야 한다.
- 변환 결과는 원문의 핵심 의도, 제약 조건, 고유명사, 출력 형식을 보존해야 한다.
- 이 최적화 과정에는 별도의 유료 클라우드 번역 API가 사용되면 안 된다.
- 사용자는 기본 모드에서 내부 최적화 과정을 강제로 보지 않아야 한다.
- 최적화 모드는 `기본`, `절감 우선`, `품질 우선`, `긴 컨텍스트`를 지원해야 한다.

### FR-04. 클라우드 모델 호출

- 최적화된 영어 프롬프트가 선택된 클라우드 모델로 전달되어야 한다.
- 기본 모델을 설정할 수 있어야 한다.
- 모델 호출 실패 시 사용자에게 명확한 오류 상태와 다음 안전한 행동이 보여야 한다.
- 실패한 경우에도 원문 프롬프트, 작업 상태, 이전 응답은 유지되어야 한다.

### FR-05. 한국어 응답 복원

- 클라우드 모델의 영어 응답은 로컬 LLM을 통해 자연스러운 한국어로 복원되어야 한다.
- 최종 응답은 사용자가 바로 읽고 활용할 수 있는 한국어 품질이어야 한다.
- 숫자, 표, 체크리스트, 구조화된 서식, 핵심 정보는 복원 과정에서 손상되면 안 된다.

### FR-06. 절감 효과 계산과 메타 정보

- 원문 한국어 기준 예상 토큰 수와 최적화 영어 기준 예상 토큰 수를 비교할 수 있어야 한다.
- 요청별 절감률, 사용 모델, 지연 시간 메타 정보가 응답 단위로 표시되어야 한다.
- 누적 절감 토큰과 절감 비용은 히스토리 또는 대시보드에서 볼 수 있어야 한다.
- 절감 효과는 전문 용어보다 사용자 친화적 문구를 우선 사용해야 한다.

### FR-07. 최적화 히스토리와 고급 미리보기

- 사용자는 원문 한국어 프롬프트, 최적화된 영어 프롬프트, 최종 한국어 응답을 필요할 때 펼쳐볼 수 있어야 한다.
- 각 단계에는 토큰 수와 절감률이 표시되어야 한다.
- 고급 영어 프롬프트 미리보기를 끄면 일반 사용자는 최종 답변 위주 경험을 유지해야 한다.
- 히스토리 상세는 제품 차별점을 설명하는 증거 화면이어야 한다.

### FR-08. 작업대 멀티채팅

- 작업대는 최소 2개 이상의 작업 패널을 동시에 보여야 하며, 기본 방향은 2x2 그리드다.
- 좌측에는 최근 작업 레일이 있어야 한다.
- 사용자는 작업 카드를 특정 패널로 열어 같은 작업을 이어갈 수 있어야 한다.
- 각 패널은 독립적인 대화 기록, 상태, 활동 로그, 추가 지시 입력창을 가져야 한다.
- 빈 패널은 `작업을 끌어오세요` 또는 `새 채팅` 같은 명확한 CTA를 보여야 한다.

### FR-09. 칸반 작업 관리

- 각 AI 작업은 카드로 표현되어야 한다.
- 컬럼은 최소 `기획`, `진행 중`, `AI 검토`, `사람 검토`, `완료`를 포함해야 한다.
- 사용자는 카드 상태를 변경할 수 있어야 한다.
- 카드에는 작업 제목, 프로젝트, 상태, 사용 도구, 최근 활동이 보여야 한다.
- 카드에서 해당 작업을 작업대 또는 상세 대화로 열 수 있어야 한다.

### FR-10. 프로젝트 관리

- 사용자는 프로젝트를 만들고 이름, 설명, 목표를 수정할 수 있어야 한다.
- 여러 작업을 프로젝트에 연결할 수 있어야 한다.
- 프로젝트 상세에서 관련 작업, 파일, 문서, 대화 컨텍스트를 함께 확인할 수 있어야 한다.
- 프로젝트 안에서 작업 검색과 새 작업 생성이 가능해야 한다.

### FR-11. 설정

- 사용자는 기본 클라우드 모델을 선택할 수 있어야 한다.
- API 키 연결 상태를 확인하고 수정할 수 있어야 한다.
- 로컬 LLM 최적화 모드와 응답 언어를 설정할 수 있어야 한다.
- 라이트, 다크, 시스템 테마를 선택할 수 있어야 한다.

### FR-12. 데이터 보존과 화면 간 동기화

- 새로고침 또는 앱 재실행 후에도 대화, 작업, 프로젝트, 설정이 유지되어야 한다.
- 오류가 발생해도 사용자가 입력한 프롬프트와 작업 상태는 사라지지 않아야 한다.
- 채팅, 작업대, 칸반, 프로젝트 화면의 작업 상태는 서로 동기화되어야 한다.

### FR-13. 데스크탑 사용성

- 주요 기능은 실제 앱 화면에서 바로 사용할 수 있어야 한다.
- 여러 작업을 병렬로 관리해도 UI가 깨지지 않아야 한다.
- 정보 밀도는 데스크탑 작업에 맞아야 하며, 한 화면에서 작업 상태를 빠르게 파악할 수 있어야 한다.
- 모바일 대응보다 데스크탑 레이아웃 안정성이 우선이다.

## 8. UX and Screen Requirements

### 8.1 Global Navigation

- 모든 주요 화면으로 한 번에 이동할 수 있어야 한다.
- 현재 위치와 작업 맥락이 명확해야 한다.
- 장기 작업 관리 제품답게 채팅과 작업 관리 화면 간 전환 비용이 낮아야 한다.

### 8.2 Chat Screen

- 첫 화면은 채팅 중심 인박스 메타포를 분명하게 보여야 한다.
- 환영 메시지는 제품 목적과 첫 행동을 동시에 설명해야 한다.
- 추천 작업 카드는 예시 프롬프트를 즉시 입력하는 역할을 해야 한다.
- 입력창 주변에 모델 선택과 최적화 모드 선택이 배치되어야 한다.
- 응답마다 모델명, 지연 시간, 절감률, `작업대에서 계속하기` 행동이 보여야 한다.

### 8.3 Workbench Screen

- 멀티패널 관리가 이 화면의 주 역할로 즉시 이해되어야 한다.
- 최근 작업 레일과 패널 배치가 분리되어, 사용자가 작업 선택과 실제 대화를 동시에 수행할 수 있어야 한다.
- 패널별 상태와 대화가 섞이지 않아야 한다.

### 8.4 Kanban Screen

- 작업 흐름 상태를 한눈에 볼 수 있어야 한다.
- 칸반은 실제 대화 화면이 아니라 흐름 관리 화면이라는 차이가 분명해야 한다.
- 카드에서 작업대 열기 행동이 빠르게 보여야 한다.

### 8.5 Projects Screen

- 장기 작업 묶음이라는 개념이 명확해야 한다.
- 프로젝트 상세는 작업, 파일, 문서, 대화 컨텍스트를 함께 보여주는 허브여야 한다.

### 8.6 Usage Dashboard and History

- 제품 차별점인 비용 절감 가치가 가장 잘 드러나는 화면이어야 한다.
- 비교 관점은 `최적화 없이 사용했다면` 대 `최적화 적용 후`여야 한다.
- 일반 사용자 모드와 고급 사용자 모드의 정보 노출 차이를 지원해야 한다.

### 8.7 Settings Screen

- 모델, API 키, 최적화, 언어, 테마를 하나의 설정 영역에서 관리할 수 있어야 한다.
- 연결 상태와 변경 가능 항목이 모호하지 않아야 한다.

## 9. State Contract

### Empty State

- 첫 진입 시 무엇을 먼저 해야 하는지 설명해야 한다.
- 추천 작업 카드 또는 명확한 입력 CTA가 보여야 한다.

### Loading State

- 요청 처리 중 `로컬 최적화`, `모델 응답 대기`, `한국어 복원` 같은 사용자 친화적 진행 맥락이 제공되어야 한다.
- 작업이 멈춘 것처럼 보이면 안 된다.

### Error State

- 오류는 사용자 언어로 설명되어야 한다.
- 다음 안전한 행동이 제시되어야 한다. 예: 재시도, 설정 확인, 다른 모델 선택.
- 오류가 나도 기존 입력과 작업 상태는 유지되어야 한다.

### Success State

- 응답 완료 시 무엇이 생성되었는지와 절감 효과가 함께 확인되어야 한다.
- 작업이 현재 어느 화면과 상태에 속하는지 업데이트가 보여야 한다.

### Completed State

- 사용자가 작업을 완료로 넘겼을 때 칸반과 프로젝트, 히스토리에서 완료 상태가 일관되게 보여야 한다.

## 10. Acceptance Criteria by Requirement Area

### AC-01. 첫 화면과 채팅 시작

- 비개발 사용자도 첫 화면에서 이 앱이 `한국어로 AI 작업을 시작하고 관리하는 데스크탑 앱`임을 이해할 수 있어야 한다.
- 첫 화면에서 사용자는 다음 행동을 추측하지 않고 바로 첫 요청을 입력하거나 추천 작업을 고를 수 있어야 한다.
- 첫 화면은 일반 랜딩 페이지나 차가운 기본 대시보드처럼 보이면 안 된다.

### AC-02. 한국어 최적화 루프

- 사용자는 한국어 원문을 그대로 남긴 채 응답을 받을 수 있어야 한다.
- 최적화된 영어 프롬프트는 내부적으로 사용되더라도 원문 의미와 구조 보존에 실패하면 안 된다.
- 한국어 복원 결과가 어색하거나 구조 정보를 잃어서는 안 된다.

### AC-03. 절감 효과 가시화

- 각 요청에서 절감률을 사용자 친화적으로 볼 수 있어야 한다.
- 누적 절감량과 절감 비용은 히스토리나 대시보드에서 신뢰 가능하게 조회되어야 한다.
- 고급 미리보기는 켜고 끌 수 있어야 하며, 기본 경험을 과도하게 복잡하게 만들면 안 된다.

### AC-04. 장기 작업 확장

- 채팅 중인 작업을 작업대, 칸반, 프로젝트로 자연스럽게 확장할 수 있어야 한다.
- 같은 작업을 어느 화면에서 보더라도 제목, 상태, 최근 활동, 대화 이력이 일관되어야 한다.
- 작업대는 실제로 여러 작업을 동시에 관리하는 데 충분한 정보 밀도와 패널 독립성을 가져야 한다.

### AC-05. 안정성과 지속성

- 앱 재실행이나 오류 후에도 사용자는 작업을 이어갈 수 있어야 한다.
- 모델 호출 실패는 사용자에게 설명 가능하고 복구 가능한 상태로 남아야 한다.
- 화면 간 동기화 실패로 인해 서로 다른 상태가 보이면 안 된다.

## 11. Validation Plan

### Scenario Validation

1. 첫 진입 사용자가 첫 화면만 보고 제품 목적과 첫 행동을 설명할 수 있는지 확인한다.
2. 한국어 장문 요청을 입력해 원문 보존, 로컬 최적화, 모델 호출, 한국어 복원이 한 흐름으로 이어지는지 확인한다.
3. 모델 호출 실패를 강제로 발생시켜 입력과 작업 상태가 유지되고 오류 안내가 이해 가능한지 확인한다.
4. 채팅에서 시작한 작업을 작업대로 넘기고, 칸반 상태를 바꾸고, 프로젝트에 연결한 뒤 모든 화면에 동기화되는지 확인한다.
5. 사용량 대시보드와 히스토리에서 요청별 절감률과 누적 절감량을 확인할 수 있는지 검증한다.
6. 고급 영어 프롬프트 미리보기 토글을 껐을 때 일반 사용자가 복잡성을 느끼지 않는지 확인한다.
7. 앱 재실행 후 대화, 작업, 프로젝트, 설정이 복원되는지 확인한다.

### UX Validation

1. 첫 화면이 채팅 중심 인박스 메타포를 분명하게 보여주는지 검토한다.
2. 결과 UI가 기본 카드 대시보드의 재색칠 수준이 아닌지 검토한다.
3. 화이트 기반, 파란 포인트, 민트 절감/성공 컬러가 유지되면서도 차갑지 않은지 검토한다.
4. 데스크탑에서 2개 이상 작업을 병렬로 열어도 레이아웃 안정성이 유지되는지 확인한다.

### Copy Validation

1. 버튼과 메시지가 `Submit` 같은 일반 용어 대신 제품 맥락에 맞는 문구를 사용하는지 확인한다.
2. 빈 상태, 오류 상태, 성공 상태 문구가 각각 다음 행동을 분명히 안내하는지 확인한다.

## 13. Must-Avoid Notes

1. 첫 화면을 마케팅 랜딩이나 일반 카드 대시보드처럼 만들지 않는다.
2. 기본 사용자 경험에서 영어 프롬프트 최적화 과정을 개발자 도구처럼 과도하게 노출하지 않는다.
3. 절감 효과를 숨기거나 추상적 메시지로만 표시하지 않는다.
4. 긴 작업을 하나의 채팅 스레드에만 고정해 작업대, 칸반, 프로젝트 확장을 막지 않는다.
5. 오류 시 사용자 입력, 작업 상태, 이전 대화를 잃게 만들지 않는다.
6. 화면마다 같은 작업이 다른 상태로 보이게 만들지 않는다.

## 14. Risks and Decisions Needed

### Product Risks

1. 로컬 LLM의 한국어 의도 보존 품질이 낮으면 비용 절감보다 결과 품질 저하가 더 크게 느껴질 수 있다.
2. 절감 비용 계산이 지원 모델별 가격 체계와 다르게 보이면 사용자 신뢰를 잃을 수 있다.
3. 클라우드 모델별 응답 형식 차이로 인해 한국어 복원 품질과 메타 정보 표기가 불안정해질 수 있다.
4. 멀티채팅, 칸반, 프로젝트, 히스토리를 동시에 다루는 범위가 넓어 상태 동기화 실패 위험이 크다.

### Decisions Needed in Later Phases

1. 초기 릴리스에서 우선 지원할 클라우드 모델과 지원 순서를 정해야 한다.
2. 절감 비용 산정의 기준 가격표와 토큰 계산 방법을 명확히 정해야 한다.
3. 로컬 최적화 실패 시의 폴백 UX 기준을 정해야 한다.
4. 프로젝트 화면에서 `파일`과 `문서`를 어느 수준까지 포함할지 구체화해야 한다.
5. 칸반 화면을 글로벌 내비게이션의 최상위 항목으로 둘지, 프로젝트 또는 작업대에서 진입하는 보조 화면으로 둘지 결정해야 한다.

### Open Questions

- 현재 사용자 입력 기준의 미해결 질문은 없다.
- 다만 위 결정 사항은 아키텍처 및 후속 상세 설계 전에 명시적으로 다뤄야 한다.

## 15. Final Acceptance for This PRD Phase

1. PRD는 최신 brief와 성공 기준을 모순 없이 반영한다.
2. PRD는 구현 아키텍처를 고정하지 않으면서도 후속 아키텍처 단계가 기술 제약을 도출할 수 있을 만큼 구체적이다.
3. PRD는 목표, 비목표, 핵심 사용자 흐름, 요구사항, 성공 기준 매핑, 검증 계획, 리스크를 모두 포함한다.
4. PRD는 채팅 중심 인박스, 데스크탑 워크스페이스, 로컬 비용 최적화, 상태 동기화라는 제품 핵심을 유지한다.
