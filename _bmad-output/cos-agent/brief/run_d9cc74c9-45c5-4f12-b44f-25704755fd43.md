# Talkin AI Brief

## Problem / Opportunity
한국어 사용자는 AI를 자연스럽게 쓰려면 한국어로 입력하고 싶지만, 상용 클라우드 모델은 영어 중심 토큰 흐름에서 더 비용 효율적으로 동작하는 경우가 많다. 그 결과 사용자는 한국어 사용성을 유지하면 비용이 늘고, 비용을 줄이려면 영어 프롬프트 작성 부담을 떠안게 된다.

Talkin AI의 기회는 이 충돌을 제품이 대신 해결하는 데 있다. 사용자는 끝까지 한국어만 사용하고, 앱 내부의 로컬 LLM이 한국어 의도와 맥락을 보존한 영어 기반 최적화 프롬프트를 만들어 클라우드 모델 호출 비용과 토큰 사용량을 줄인다. 동시에 단일 채팅 앱을 넘어서 긴 작업을 관리할 수 있는 데스크탑 워크스페이스를 제공해, 한국어 사용자가 AI를 더 오래, 더 깊게 활용하게 만든다.

## Target User or Use Case
초기 타깃은 한국어로 장문 지식 작업을 반복하는 운영 담당자와 일반 지식 작업자다. 대표 작업은 사업계획서 초안 작성, 긴 PDF 요약, 문서 정리, 카피 다듬기, 후속 작업 관리처럼 한 번의 질문으로 끝나지 않는 업무형 AI 사용이다.

핵심 사용 흐름은 다음과 같다. 사용자는 채팅 화면에서 한국어로 바로 작업을 시작하고, 필요하면 같은 작업을 작업대 멀티채팅, 칸반, 프로젝트로 자연스럽게 확장한다. 이 과정에서 원문, 최적화 프롬프트, 응답, 절감 효과, 작업 상태가 끊기지 않고 이어져야 한다.

## Core Value Proposition
Talkin AI는 "한국어 AI를 더 오래, 더 깊게" 쓰게 하는 데스크탑 AI 작업 공간이다.

- 사용자는 영어를 직접 쓰지 않아도 된다.
- 로컬 LLM이 유료 번역 API 없이 한국어 입력을 영어 기반 토큰 흐름으로 최적화하고, 영어 응답을 자연스러운 한국어로 복원한다.
- 클라우드 모델 비용은 실제 추론이 필요한 핵심 작업에 집중된다.
- 결과적으로 사용자는 한국어 사용성을 유지하면서도 토큰 절감과 비용 절감 효과를 직관적으로 확인할 수 있다.
- 제품 경험은 단일 채팅창이 아니라, 채팅 중심 인박스에서 장기 작업 관리로 확장되는 데스크탑 워크스페이스여야 한다.

## Success Criteria Alignment
이 제품의 성공 기준은 "핵심 기능이 실제 제품 안에서 정상적으로 동작하는가"에 맞춘다. 초기 브리프는 아래 다섯 축에 정렬된다.

1. 한국어 기반 대화 루프
사용자는 한국어로 프롬프트를 입력하고 전송할 수 있어야 하며, 원문 한국어는 대화 히스토리에 보존되어야 한다. 긴 문장, 문서 요약, 보고서 작성 같은 장문 요청도 안정적으로 처리되어야 한다.

2. 로컬 비용 최적화와 응답 복원
클라우드 호출 전 로컬 LLM이 한국어 프롬프트를 영어 기반 최적화 프롬프트로 변환해야 하고, 핵심 의도, 조건, 고유명사, 출력 형식을 유지해야 한다. 이 과정에는 별도의 유료 클라우드 번역 API가 없어야 하며, 클라우드의 영어 응답은 핵심 정보 손실 없이 자연스러운 한국어로 복원되어야 한다.

3. 모델 호출 안정성과 설정 가능성
최적화된 영어 프롬프트는 선택된 클라우드 모델로 전달되어야 하며, 기본 모델 선택과 최적화 모드 선택이 가능해야 한다. 호출 실패 시 사용자에게 명확한 오류 상태를 보여주고, 입력과 작업 상태는 사라지지 않아야 한다.

4. 절감 효과의 가시성과 신뢰
원문 한국어 대비 최적화 영어 프롬프트의 예상 토큰 수를 비교할 수 있어야 하고, 요청별 절감률과 누적 절감 토큰/비용이 대시보드와 히스토리에서 보여야 한다. 사용자는 필요 시 원문, 최적화 프롬프트, 최종 한국어 응답을 단계별 토큰 수와 함께 펼쳐볼 수 있어야 하며, 일반 모드에서는 이 복잡성이 전면에 드러나지 않아야 한다.

5. 데스크탑 작업 공간 완성도
초기 화면은 마케팅 랜딩이 아니라 바로 업무를 시작하는 채팅 중심 인박스여야 한다. 글로벌 내비게이션에는 채팅, 작업대, 프로젝트, 사용량 대시보드/히스토리, 설정이 포함되어야 한다. 작업대는 최소 2개 이상 동시 작업이 가능한 멀티채팅 화면이어야 하고, 칸반은 상태 기반 작업 흐름을 보여주며, 프로젝트는 장기 작업 묶음을 관리해야 한다. 새로고침 또는 앱 재실행 후에도 대화, 작업, 프로젝트, 설정은 유지되고 서로 동기화되어야 한다.

## Initial Scope and Non-Goals
초기 범위는 아래에 집중한다.

- 한국어 입력 → 로컬 최적화 → 클라우드 추론 → 한국어 복원으로 이어지는 핵심 대화 플로우
- 채팅, 작업대, 프로젝트, 사용량 대시보드/히스토리, 설정으로 이어지는 데스크탑 워크스페이스 구조
- 요청별/누적 토큰 및 비용 절감 가시화
- 기본/절감 우선/품질 우선/긴 컨텍스트 최적화 모드
- 작업 상태 보존, 오류 복구, 화면 간 상태 동기화
- 화이트 기반 SaaS 톤 위에 파란 포인트와 민트 절감 색상을 사용하는 실제 업무형 앱 화면

이번 단계의 비목표는 아래와 같다.

- 모바일 우선 경험 설계
- 범용 에이전트 마켓플레이스나 대규모 외부 툴 생태계 확장
- 로컬 LLM 종류, Translation MCP 세부 계약, 저장소 구조, 데이터 아키텍처 같은 구현 설계 확정
- 마케팅 페이지 중심의 브랜딩 작업

## Explicit Uncertainty Notes
- 어떤 로컬 LLM이 한국어 의도 보존과 비용 절감 간 균형을 가장 잘 내는지는 후속 검증이 필요하다.
- 절감 비용 산정 방식은 지원 모델별 가격 체계와 토큰 계산 기준을 이후 단계에서 명확히 해야 한다.
- GPT, Claude, Gemini 지원 범위는 동일 UX를 유지하되, 실제 초기 릴리스에서의 우선 연동 순서는 추후 결정이 필요하다.
