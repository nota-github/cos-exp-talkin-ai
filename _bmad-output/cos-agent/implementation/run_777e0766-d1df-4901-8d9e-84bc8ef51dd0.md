## Implementation Summary

- The renderer now derives a user-facing run-feedback card from the persisted run summary instead of showing only a generic queued label. The card stays inside the chat lane, uses three compressed step pills, and changes copy across local optimization, model wait, Korean restoration, completion, and failure states.
- Failure handling now reads persisted run-stage details from SQLite-backed query results so the chat feed can explain where the run stopped and offer the safest next action first. Auth failures push users toward settings, retryable provider failures expose retry plus model choice, and local-engine failures avoid surfacing raw English error codes.
- `retryRun` now creates a brand-new queued run that points to the same stored source message instead of asking the user to retype the Korean prompt. The preserved assistant response stays in the conversation, and the renderer can also prefill the same Korean source into the composer when the user wants to switch models before resubmitting.
