# Selected Story

- Story Key: `story-5.6`
- Title: `Project list, editor, and task linking`

## Quality Plan

| Criterion ID | Intended Evidence |
| --- | --- |
| `story-5.6:GOAL-1` | Add a persistent project-management slice that lets the desktop app create/edit projects and link or unlink canonical tasks from a project. |
| `story-5.6:SCOPE-1` | Replace the primary `/projects` surface with a project-list-first hub view while keeping the existing Kanban flow available as a secondary work-management view. |
| `story-5.6:SCOPE-2` | Add typed IPC commands and persistent service logic for project create/edit, then wire a renderer form that edits `name`, `description`, and `goal`. |
| `story-5.6:SCOPE-3` | Add one typed task-project linking command with `projectId` nullable for link and unlink flows, backed by canonical `tasks.project_id`. |
| `story-5.6:AC-1` | Persist project creation and updates through SQLite-backed repositories and expose them in the `/projects` editor UI. |
| `story-5.6:AC-2` | Persist task link/unlink mutations and reflect the canonical project assignment in related projections. |
| `story-5.6:AC-3` | Expose a project list query ordered by one effective activity timestamp, defined as the newer of recent task activity and project update time, and render that ordering in the list UI. |
| `story-5.6:AC-4` | Emit invalidation targets for project/task mutations so project list/detail and related task projections can refetch from one source of truth. |
| `story-5.6:AC-5` | Make the `/projects` screen read like a long-running work hub entrance rather than a settings form, using project list, editor, task-linking context, and a secondary Kanban route/tab affordance. |
| `story-5.6:VAL-1` | Add automated persistence coverage that creates and edits a project after linked task activity, recreates the service, and confirms the updated values still survive restart and reorder correctly. |
| `story-5.6:VAL-2` | Add automated coverage for linking a task into a project and unlinking it again, including projection-visible effects. |
| `story-5.6:VAL-3` | Add automated surface-state/source checks for empty and populated project list UI states; record guided live verification separately if browser evidence is unavailable. |
| `design-must-have-7` | Keep `/projects` aligned to the approved `프로젝트` destination in global navigation. |
| `design-must-have-10` | Preserve clear first-action copy on the project hub so empty-state users know to create a project or connect an existing task. |
| `design-reference-1` | Keep the Korean-first desktop workspace framing and treat project management as an extension of the same long-running AI work item model. |
| `design-layout-1` | Keep Kanban/flow management secondary to the project-management hub entrance instead of the only visible `/projects` surface. |

## In Scope for This Run

- Added a persistent `projects` main-process service with:
  - project list query ordered by task activity or project update time
  - project detail query with linked task summaries
  - project create/edit commands
  - one nullable `setTaskProject` command for link and unlink
- Extended shared IPC contracts, preload bridge, query invalidation wiring, and Electron startup to expose the new project-management slice.
- Rebuilt `/projects` as a project-hub-first screen with:
  - recent-activity-sorted project rail
  - create/edit form for `name`, `description`, `goal`
  - linked-task and recent-task command panels
  - secondary `보조 흐름 보드` view so the earlier Kanban slice remains available without being the only surface
- Added story-specific automated coverage for restart persistence, task link/unlink projection sync, and empty/populated project-hub UI states.
- Patched the remaining ordering bug so populated projects now sort by the newer of hub edits and linked-task activity in both persistent and in-memory list builders.

## Deferred Work

- `story-5.7` project detail hub, in-project search, and richer file/context sections remain out of scope.
- No new Usage/History coupling was added for retroactive `usage_category` reclassification when an old task is later linked to a project.
- Live browser verification of the real `/projects` route is still required if reviewers want screenshot-grade evidence, because the local renderer could not be launched in this workspace.
- `story-5.6:AC-5` and `story-5.6:VAL-3` still need orchestrator-prepared live `/projects` screenshots from a runnable renderer environment; this run could only re-confirm the local runtime blocker.
